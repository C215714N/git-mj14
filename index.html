Areas de GIT

Working Directory

El area trabajo corresponde a la carpeta donde creamos el repositorio y basicamente vamos a crear, editar o eliminar archivos o subcarpetas. En muy pocas ocasiones utilizaremos la consola y solamente necesitamos implementar el comando de GIT que iniciliza el repositorio.

Staging Area (INDEX)

El area de control de cambios corresponde a un archivo dentro de la carpeta .git/ que gestiona a que archivos le realizamos cambios. Tambien se encarga de realizar la capturas de codigo (SNAPSHOT) para su posterior confirmacion cuando hayamos finalizado con el trabajo correspondiente.

Repository (local/remote)

El almacen de cambios corresponde a una estructura de carpetas dentro de la carpeta .git/ que contiene las confirmaciones realizadas en archivos BLOB (Binary Large Object). Se utilizan principalmente como referencia para comparar, visualizar y modificar el contenido de nuestro proyecto.

Apuntadores GIT

HEAD

Apuntador Movil que nos indica donden nos encontramos actualmente y permite desplazarnos dentro del historial de confirmaciones. Se utiliza como referencia al momento de volver hacia atras dentro del historial de cambios, ademas de realizar comparaciones entre los commits y nuestro working directory.

BRANCH

Apuntador Dinamico que se corresponde con el ultimo commit de la historia en la que se encuentre. Se utiliza para implementar diferentes caracteristicas o resolver conflictos con respecto al codigo del proyecto, sin comprometer la linea de tiempo principal. Sirven como referentes para volver si nos encontramos en algun commit anterior dentro del historial de confirmaciones.

STASH

Apuntador de la zona temporal de cambios que se utiliza cuando necesitamos desplazarnos en el historial de confirmacion y poseemos cambios en nuestro proyecto que no ameritan un commit. Los cambios se apilan apuntando al commit de la rama donde se realizo y su INDEX, para luego reaplicarlos realizando una operacion merge a la inversa.

TAG

Apuntador Estatico que hace referencia al commit donde se haya aplicado. Se utiliza para identificar la version del proyecto y normalmente se utilizan valores como vn.s.r donde "n" corresponde a la version principal, "s" a la subversion (caracteristicas o implementaciones) y "r" a la revision de errores o correcciones de los mismos.
